## What this is

Peridio makes Avocado OS, an immutable embedded Linux runtime shipped as a binary distribution: a read-only A/B root filesystem, composed sysext/confext extensions, and a writable `/var`. Everything about a build lives in one declarative file, `avocado.yaml` — the target, the runtimes, the extensions and packages that compose the image.

This note is the pose-detection demo from the NXP "Engineer's Guide to Simplifying Embedded Linux Development" webinar, written down so you can reason about it and reproduce it. It runs [MoveNet](https://www.kaggle.com/models/google/movenet) pose estimation on the i.MX 8M Plus, doing inference on the board's integrated NPU, with frames coming off a plain USB webcam. It is one of our reference examples — `imx8mp-npu-pose` — and the interesting claim is not "AI runs on the 8MP." It is that the exact same project retargets to the other NPUs in the i.MX line without touching your application.

## The problem: every NPU speaks a different dialect

Edge AI silicon has gone wide. NXP alone ships the 8M Plus with one NPU, the i.MX 93 with another, the 95 with a larger one, and discrete accelerators above that. They scale in performance — and they also vary in the exact incantation you need to get a model to run on them. Quantization, operator support, the output format the accelerator will actually load: all of it shifts per part.

What that does to a team is predictable. You end up with a loose bag of scripts that carries a model from your training team's output into something each specific NPU will accept, and that bag is different for every board. The moment you want to evaluate a cheaper part, scale up because the model got heavier, or move because a part went short in the supply chain, you are re-deriving that pipeline by hand. The hardware swap is easy. The software underneath it turned a part change into a project.

That is the friction this demo is built to remove.

## The finale first: pose detection on the 8MP NPU

On screen is a live camera feed with a skeleton overlaid on it — shoulders, elbows, wrists tracked in real time. That is MoveNet doing pose estimation, and every frame of inference is running on the NPU on the i.MX 8M Plus, not the CPU. The camera is a USB webcam plugged straight into the dev kit; the board serves the annotated feed over HTTP, which is why you are seeing it in a browser pointed at the device.

![The full demo: VS Code driving the reference on the left, the live NPU pose feed in the center, the i.MX 8M Plus running it on the right.](/img/field-notes/imx8mp-npu-pose/02-demo-layout.jpg)

The pipeline is ordinary embedded plumbing, which is the point: GStreamer pulls frames from the UVC camera, hands them to a small Python app that runs the quantized model on the NPU and draws the keypoints, and Weston renders the output. All of it is declared in the config, not hand-installed on the device.

## Anatomy of the config

The whole project is one `avocado.yaml`. From the top it names the targets it supports and the default to build, then composes a runtime out of extensions. An extension is a reusable fragment of an image — you develop it once and share it across projects. Some are inline (your app, your config); some are shared, pulled from Avocado, from Peridio, or from your own team.

The board support package is templated on the active target, which is the hinge the whole portability story turns on:

```yaml
cli_requirement: '>=0.41.0'

default_target: 'imx8mp-evk'

supported_targets:
  - imx8mp-evk
  - imx8mp-frdm
  - imx8mp-var-dart
  - ucm-imx8m-plus
  - imx93-evk
  - imx93-frdm
  - imx93-var-dart

distro:
  release: 2024
  channel: edge-next

runtimes:
  dev:
    extensions:
      - avocado-ext-dev
      - avocado-ext-sshd-dev
      - avocado-bsp-{{ avocado.target.board }} # resolves per target
      - config
      - app
```

![The config names the target once. `supported_targets` spans the 8MP and i.MX 93 families, and the BSP line templates on `{{ avocado.target.board }}`.](/img/field-notes/imx8mp-npu-pose/04-config-supported-targets.jpg)

The application extension installs exactly what this demo needs and nothing else — that is the whole security-footprint argument for an immutable OS. You are adding lines, not images:

```yaml
extensions:
  app:
    packages:
      # --- GStreamer: the capture + inference + render pipeline ---
      gstreamer1.0: '*'
      gstreamer1.0-plugins-base-videoconvert: '*'
      gstreamer1.0-plugins-base-app: '*'
      gstreamer1.0-plugins-good-video4linux2: '*'
      gstreamer1.0-plugins-good-videocrop: '*'
      gstreamer1.0-plugins-good-cairo: '*'
      gstreamer1.0-plugins-good-jpeg: '*'
      gstreamer1.0-plugins-bad-waylandsink: '*'

      # --- Python: drive the pipeline + draw the skeleton ---
      python3: '*'
      python3-pygobject: '*'
      python3-pycairo: '*'
      python3-numpy: '*'

      # --- Camera + display ---
      kernel-module-uvcvideo: '*' # my USB webcam Just Works
      v4l-utils: '*'
      weston: '*'
      weston-init: '*'
```

![The `app` extension: GStreamer, a little Python, the UVC kernel module for the webcam, and Weston — declared as package lines, not a hand-built image.](/img/field-notes/imx8mp-npu-pose/05-config-app-packages.jpg)

Two things worth calling out. Kernel support like UVC video is a package line, not a kernel rebuild — Avocado builds only what is required to boot into the kernel itself and ships the rest as modules you pull in. And the SDK is opt-in per language: this reference pulls in `nativesdk-uv` so `uv` can drive the in-SDK pip install of TensorFlow used for INT8 quantization — and, importantly, TensorFlow is build-time-only; it is not installed on the target. The same environment cross-compiles C, C++, Rust, Elixir, Java. You are not downloading hundreds of gigabytes of toolchain to build one app; you pick exactly what you need.

## Enabling the NPU is a shared extension, not a bag of scripts

Here is the part that matters. The `app` extension declares itself as a MoveNet model that has been quantized for the target NPU:

```yaml
  app:
    types:
      - sysext
      - confext
    version: '0.1.0'
    summary: 'MoveNet INT8 single-person pose'
    overlay: app/overlay
    packages:
      # MoveNet is quantized to per-tensor INT8 for the NPU
      # the .tflite is copied into the extension overlay
```

The reference does this work in the open — `fetch-model.sh` pulls the model and `quantize-model.py` runs it through `app-compile.sh` to produce a per-tensor INT8 `.tflite` the 8MP's NPU will load — so you can see exactly what happens.

![The `app` extension is a `MoveNet INT8 single-person pose` model, quantized for the NPU and dropped into the overlay.](/img/field-notes/imx8mp-npu-pose/06-config-movenet-extension.jpg)

But in practice you don't own that step. There is a shared Avocado extension that knows how to quantize a model input _for a given i.MX NPU_. Point it at the 8MP and it produces an 8MP-shaped model. Point it at the 93 — because that NPU differs — and it produces the model that accelerator expects instead. The "magic words" per NPU live in an extension we maintain, not in your repo.

That is why enabling the NPU and staying portable across NPUs are the same feature. The thing that gets your model onto the 8MP is the same mechanism that re-targets it to a different NPU.

## The target swap

To move this application to a different i.MX NPU, you change the target. The BSP template resolves to the new board's support package, the feed serves that architecture's pre-built binaries, and the quantization extension re-emits the model for the new NPU. Your application source does not move:

```yaml
default_target: 'imx93-evk'
```

or, without editing the file, pass it on the command line:

```bash
avocado build --target imx93-frdm
```

This reference is configured for the 8MP and i.MX 93 families today (including the Variscite `var-dart` SOMs and the `ucm-imx8m-plus` module); the same mechanism extends up the line as you add targets. It is not a rebuild-the-world event either. We build the world of packages ahead of time, per architecture — tens of thousands of them — so a target change pulls the right pre-built packages down at install time and `avocado build` cross-compiles your pieces into the image. The work is already done on the other side of the swap.

## Three commands, then it's on the board

From the config to a running device is the same three commands on every target:

```bash
avocado install # pull the pre-built packages for this target
avocado build     # compose the runtime, cross-compile the app + model
avocado provision # stream it onto the board
```

<PullQuote>
  `avocado build` composed the full `dev` runtime — app, BSP, dev and sshd extensions, config,
  images, and the runtime — in **1 minute 10 seconds**.
</PullQuote>

![`avocado build` composing the full runtime from pre-built packages in 1m 10s.](/img/field-notes/imx8mp-npu-pose/07-avocado-build.jpg)

Provisioning is handled by profiles that know each board's flash path. The 8MP has an on-board eMMC (what we used) and an SD cage; the `uuu-emmc` profile puts the part into serial-programming mode per the boot-switch chart and streams the image down, the `sd` profile burns a card instead. You can blend profiles or write your own.

## Iterating without a re-flash

Once the device is up, you don't re-burn it to change something. `avocado control` on the device shows status and which extensions are loaded — here the BSP, config, and app extensions are all merged and active:

![`avocado control` on the device: the BSP, config, and app extensions loaded and merged.](/img/field-notes/imx8mp-npu-pose/08-avocado-control.jpg)

Add a package or tweak the app and `avocado deploy` streams just the change down and merges it in place, no re-flash and no reboot. Under the hood the boot chain is A/B and minimal — Avocado only builds what is needed to boot into the root filesystem, initramfs, and kernel, so the A/B partitions stay small and rarely change size, while the rest of the disk is a Btrfs `/var` holding the extensions. That is what makes an in-place update cheap instead of a full re-image.

## A note on the agentic path

One of the honest pains of this work is not knowing the package you need for a given peripheral. Our tools ship an MCP server that can query the package feeds, make the choice, and update your `avocado.yaml` in line — usable from Claude Code or Codex, or our own integrated tooling. It is a convenience layer over the same config, not a separate magic system; the file it edits is the file above.

## Issues Encountered

- **Different NPUs, different results — validate per part.** The config retargets and the quantization extension re-emits the model, but a heavier NPU is not just "the same, faster." Operator coverage and quantization behavior differ, so accuracy and frame rate should be re-measured on each target rather than assumed to carry over. Portability gets you a running model on the new NPU; it does not excuse you from validating it.
- **Model in, model out is still your loop.** The shared extension handles the NPU-specific quantization, but choosing the source model and confirming keypoint quality is on you. The value is that the per-board plumbing stops being your problem, not that model selection does.
- **Deploy is live plumbing, and live plumbing hiccups.** In the webinar the in-place `avocado deploy` hit a transient slot-detection error on the first attempt — the honest reality of a live demo. The build and the running device are the load-bearing claims here; the streaming update is a convenience on top.

## Reproduce it

[ENGINEER: Once the reference is public, add the `reference` prop to `<TestStatus>` and confirm the init command below.]

Initialize the project, build the 8MP pass end to end, and watch pose detection come up off a USB camera:

```bash
avocado init --reference imx8mp-npu-pose imx8mp-npu-pose
cd imx8mp-npu-pose
avocado install -f
avocado build
avocado provision -r dev # uuu-emmc on the 8MP
```

Then change `default_target` to `imx93-evk` (or pass `--target imx93-frdm`), rerun `install` / `build` / `provision`, and diff `avocado.yaml` to confirm one line moved. The application source is untouched; the NPU underneath it is not.
